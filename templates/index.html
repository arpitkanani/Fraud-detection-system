Pipeline in five clear steps

Good for explaining the project to both technical and non-technical viewers.

01 data intake

Fast loading and merge

Transaction and identity files are loaded efficiently with PyArrow and merged on TransactionID using a left join.
02 features

Fraud-focused signal creation

The pipeline creates time, amount, email, device, card, address, identity, and V-column aggregate features before scoring.
03 preprocessing

Saved transformer reuse

Categorical, frequency, and numeric transformations are applied with the same saved preprocessor used during training.
04 model choice

Best model wins

LightGBM, XGBoost, and CatBoost are compared, and the best ROC-AUC model becomes the production model.pkl.
05 output

Review-ready result table

The final output includes S.No, TransactionID, prediction, fraud probability, and risk factor, plus a downloadable CSV.
Bonus

Interactive review layer

In the detector page, clicking High, Medium, or No Risk loads matching TransactionIDs from the saved prediction file.

What the project expects

The structure stays friendly for real-world uploads while still following the IEEE-CIS schema.

Merged mode

One file in, one prediction file out

Upload a single merged CSV when your transaction and identity data are already joined. Required columns are TransactionID, TransactionAmt, and card1.

Best for ready-to-score batches.
Optional columns are handled safely if missing.
Separate mode

Transaction and identity handled together

Upload transaction.csv and identity.csv separately when they still exist as two source files. The app merges them with a left join on TransactionID.

Identity file only needs TransactionID for the join.
Both id_01 and id-01 naming styles are accepted.
